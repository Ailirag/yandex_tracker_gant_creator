# Интеграция с 1С: график доступности ресурса

Два пути отдать планировщику доступность групп СА и разработчиков.

## Путь A — ручная выгрузка запросом (доступно сейчас)

Файл [`Запрос_ГрафикДоступности.txt`](Запрос_ГрафикДоступности.txt) — готовый
запрос на языке 1С. Основан на проверенном запросе «график доступного ресурса»,
изменён только финальный разрез (агрегация по роль-группе + Роль и Направление).

Как использовать:
1. Открыть **Консоль запросов** в 1С, вставить запрос.
2. Задать параметры `&НачалоПериода` и `&ОкончаниеПериода` (тип Дата), например
   сегодня .. конец горизонта планирования.
3. Выполнить, сохранить результат как **`.xlsx`** (или `.csv`).
4. Передать планировщику:
   ```
   planner plan --capacity file --capacity-file <файл.xlsx>
   ```

**Контракт колонок** (по заголовку, порядок не важен): `Дата`, `Группа`,
`ДоступныйРесурс` — обязательны; `Роль`, `Направление` — опциональны
(игнорируются планировщиком). Имена групп в выгрузке уже совпадают с маппингом
команд в `config.py`. FTE берётся как есть — отпуска и календарь учтены в 1С;
резерв на поддержку/fast-track моделируется в 1С отдельными «Событиями расхода
ресурсов», в планировщике коэффициенты к файлу НЕ применяются.

## Путь B — HTTP-сервис (автоматизация, позже)

Назначение: отдать планировщику живой график доступности ресурса групп
(остатки РН «ДоступностьРесурса» с учётом отпусков, резервов и производственного
календаря) и принять результат расчёта для хранения/анализа в 1С.

## Объекты метаданных

1. **HTTPСервис `ПланировщикЗадач`**
   - Корневой URL: `planner`
   - Шаблон URL `capacity` (шаблон: `/capacity`), метод `GET`, обработчик `КапаситиGET`
   - Шаблон URL `plan` (шаблон: `/plan`), метод `POST`, обработчик `ПланPOST`

2. **РегистрСведений `РассчитанныйПланЗадач`** (новый, для приёма расчёта)
   - Измерения: `Задача` (СправочникСсылка.Задачи), `ДатаРасчета` (Дата)
   - Ресурсы: `ДатаНачала` (Дата), `ДатаОкончания` (Дата), `ПлановаяДатаЗавершения` (Дата)
   - Реквизиты: `Предупреждения` (Строка), `ИсточникРасчета` (Строка)

Публикация на веб-сервере штатная (меню администрирования), аутентификация —
базовая или ключ в заголовке за обратным прокси; сервис читает только
агрегированные FTE, персональных данных в ответе нет.

## Контракт

### GET /planner/hs/planner/capacity?from=2026-07-15&to=2026-12-31

Ответ `200 application/json`:

```json
{
  "from": "2026-07-15",
  "to": "2026-12-31",
  "groups": {
    "СА направления финансы (1С)": {"2026-07-15": 5.6, "2026-07-16": 5.6},
    "Разработчики 1С УТ-БП": {"2026-07-15": 6.3}
  }
}
```

Дни отсутствуют в ответе = 0 FTE (выходной/нет ресурса). Ключ группы —
`Наименование` группы справочника «Сотрудники».

### POST /planner/hs/planner/plan

Тело — массив объектов `{key, new_start, new_end, new_planned_completion, warnings[]}`
(файл `plan_*.json` планировщика как есть). Сервис пишет строки в
РС «РассчитанныйПланЗадач» по ключу задачи. Ответ: `{"accepted": N, "not_found": [...]}`.

## Модуль HTTP-сервиса (BSL)

Код — в файле `МодульHTTPСервиса.bsl` рядом. Запрос ёмкости повторяет
проверенный запрос «графика доступности ресурса» (остатки и обороты по дням
в иерархии сотрудников, итог по группе-родителю), поэтому цифры совпадут
с обработкой «ГрафикДоступностиРесурса».

## Проверка после вставки

1. Опубликовать базу с включённым HTTP-сервисом.
2. `curl "http://<host>/<base>/hs/planner/capacity?from=2026-07-15&to=2026-08-15"`
3. Сравнить пару групп/дней с отчётом «График доступности ресурса».
4. Запустить планировщик: `planner plan --capacity onec --onec-url http://<host>/<base>/hs/planner`
