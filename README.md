# Sprint Planner — автопланирование задач очереди ONE

Расчёт `start` / `end` / «Плановой даты завершения» для задач очереди ONE
по важности, с учётом FTE-ресурса групп «роль × направление» и графика
релизов. **По умолчанию — только расчёт (dry-run), в Трекер ничего не
пишется.**

> 🚀 **Впервые здесь / нетехнический пользователь?** Начните с пошаговой
> [ИНСТРУКЦИЯ.md](ИНСТРУКЦИЯ.md) — там всё с нуля: установка, запуск, чтение
> отчёта, безопасная запись и откат. Ниже — технический справочник.

## Запуск

```powershell
cd "D:\Claude\yandex tracker analyzer\planner"
uv run planner plan                     # расчёт, отчёты в out\
uv run planner plan --start 2026-08-01  # раскладка с конкретной даты
uv run planner plan --capacity onec --onec-url http://host/base/hs/planner
                                        # ёмкость из 1С вместо мока

# Оценка последствий (what-if): что будет, если задаче дать важность
uv run planner plan --what-if "ONE-2515=1"
#   -> консоль и лист «Смещения»: кто сдвинулся, на сколько раб. дней, из-за кого;
#      такой план НЕ становится базовым (plan_latest.json не обновляется)

# Каждый обычный прогон автоматически сравнивается с предыдущим
# (out\plan_latest.json); отключить: --no-baseline, задать явно: --baseline <файл>
```

### Отбор задач и дата старта (всё опционально, по умолчанию выключено)

```powershell
# Дата, с которой строится раскладка (по умолчанию сегодня)
uv run planner plan --start 2026-08-01

# Отсечь шум статусами (наш статус-модель, не требует знания точных имён):
uv run planner plan --exclude-in-progress   # убрать «В разработке», «Устранение замечаний»
uv run planner plan --exclude-tail          # убрать хвост: тест/ревью/готово к релизу
uv run planner plan --exclude-status "Пауза,Отложено"   # произвольные статусы по имени

# Отбор на стороне Трекера (полный контроль через язык запросов):
uv run planner plan --query '"Queue": "ONE" "Resolution": unresolved() "Важность заявки": notEmpty()'
uv run planner plan --query '"Queue": "ONE" Status: "Готово к разработке","Готово к анализу"'

# Комбинируется: чистый бэклог с конкретной даты, без задач в работе и на завершении
uv run planner plan --start 2026-08-01 --exclude-in-progress --exclude-tail
```

Исключённые задачи не планируются **и не потребляют ресурс** — это «чистый»
вид раскладки бэклога. Активные фильтры печатаются в начале прогона, а причина
исключения каждой задачи — на листе «Пропущенные». `--query` (отбор в Трекере)
и `--exclude-*` (отсев по статус-модели) дополняют друг друга: query решает,
что вообще выгружать, exclude — что из выгруженного не планировать.

**Команды без ресурса в выгрузке.** Если у команды нет ни одной группы в
источнике ёмкости (например, «ЗУП и ДО», по которой ресурс в 1С ещё не
инициализирован), все её задачи исключаются из расстановки, а в сводке
печатается предупреждение с числом задач. Поведение включено по умолчанию;
отключить — `--keep-empty-teams` (тогда такие задачи вернутся в расчёт и будут
висеть с пометкой «ёмкости не хватило»). С `--capacity mock` группы есть у всех
команд, поэтому ничего не исключается.

Результат в `out\`:
- `plan_*.xlsx` — человекочитаемый отчёт: лист «План» (расчёт и diff с
  текущими датами; изменённые ячейки подсвечены), «Пропущенные» (с причиной),
  «Инфо» (предупреждения);
- `gantt_*.html` — **диаграмма Ганта** по командам (открыть в браузере):
  свимлейны команд, задачи-полоски с фазами (СА/Dev/тест/ожидание релиза),
  маркеры релиза и дедлайна, линия «сегодня», детали по наведению;
- `plan_*.json` — машиночитаемый план (вход для `apply`);
- `backup_*.json` — текущие даты задач плана из Трекера (снимок при расчёте).

## Бэкап и восстановление (страховка перед записью)

```powershell
# 1. Снять снимок текущих дат ПЕРЕД любыми изменениями (вся очередь ONE)
uv run planner backup
#    -> out\snapshot_*.json + out\backup_latest.json

# 2. Вернуть даты «как было» из снимка. ВАЖНО: поля, которые в снимке были
#    пустыми, будут очищены в Трекере (полный возврат к исходному состоянию).
uv run planner restore --backup out\snapshot_20260715_2319.json            # всё из снимка
uv run planner restore --backup out\backup_latest.json --keys ONE-4711     # только одну задачу
# revert — псевдоним restore (для совместимости)
```

Три уровня страховки:
- `planner backup` — ручной снимок всей очереди в любой момент;
- `planner apply` **сам** делает свежий бэкап ровно изменяемых задач
  (`preapply_*.json`) непосредственно перед записью — состояние на момент
  apply, а не на момент расчёта; отключается флагом `--no-backup`;
- `backup_latest.json` всегда указывает на последний сделанный снимок.

> Очистка ранее пустых полей выполняется отправкой `null` в API Трекера —
> проверьте результат на одной задаче при первом реальном `restore`.

## Запись в Трекер (только осознанно)

```powershell
uv run planner apply --plan out\plan_20260715_1936.json --keys ONE-4711 --yes   # пробно 1 задача
uv run planner apply --plan out\plan_20260715_1936.json --limit 10             # первые 10
uv run planner apply --plan out\plan_20260715_1936.json                        # всё, с подтверждением
```

Перед записью `apply` печатает путь к свежему бэкапу и команду отката —
если что-то пошло не по плану, откат делается одной строкой.

## Модель (зафиксирована 2026-07-15)

- Фазы задачи: **СА → Dev → тест-буфер 5 раб. дней**; БА вне контура,
  ресурс тестировщиков появится в 1С позже.
- 1 FTE-день = **6 полезных часов**; на фазе задачи работает один человек
  (≤1 FTE/день), группа ведёт несколько задач параллельно в пределах ёмкости.
- Очередь: сначала задачи в статусах «В разработке»/«Устранение замечаний»,
  затем по «Важности заявки» (**меньше = важнее**), затем по дедлайну.
- **Пустая важность = не в очереди**: задача не планируется (лист
  «Пропущенные»). Исключения: уже идущая разработка (ресурс реально занят,
  помечается предупреждением) и хвостовые статусы (считается только ПДЗ).
- «Плановая дата завершения» = ближайший день «Р» системы задачи после конца
  работ (фриз — мораторий только на деплой). Системы: компоненты
  `1С УТ|1С БП|1С ЗУП|1С ДО` (+ `1С:МДМ` в плане релизов).
- Команда: компонент `Команда: …`; фолбэк — однозначные значения поля
  «Направление» (МДМ, ЗУП, ДО, Склады, Качество, Финансы). Компонента
  «Команда: МДМ» в очереди нет — МДМ ловится только фолбэком.
- Статусы: см. `src/planner/config.py` (SA_AHEAD / DEV_AHEAD /
  DEV_IN_PROGRESS / TAIL_ONLY / NOT_PLANNED / DONE).

## Источники данных

| Источник | Сейчас (MVP) | Целевое |
|---|---|---|
| Задачи, оценки, важность | REST API Трекера (токен: env `YATRACKER_TOKEN_GT`/`YATRACKER_ORGID_GT` или `.env`) | так же |
| Ёмкость групп (FTE) | `--capacity file`: выгрузка из 1С (запрос `onec/Запрос_ГрафикДоступности.txt`, отпуска учтены) ИЛИ `--capacity mock`: численность × резерв, отпуска НЕ учтены | HTTP-сервис 1С (`onec/`): те же остатки онлайн |
| План релизов | `source/План релизов 1С.xlsx`, лист 2026 | загрузить в 1С (регистр) и отдавать сервисом |

### Источник ёмкости

```powershell
uv run planner plan --capacity mock                                  # оценка по численности (по умолчанию)
uv run planner plan --capacity file --capacity-file capacity_1c.xlsx # реальная доступность из 1С (рекомендуется)
uv run planner plan --capacity onec --onec-url http://host/base/hs/planner  # онлайн из 1С (когда будет сервис)
```

Выгрузку из 1С готовит запрос `onec/Запрос_ГрафикДоступности.txt`
(колонки `Дата`, `Группа`, `ДоступныйРесурс`); подробности — в
[`onec/README.md`](onec/README.md).

## Известные ограничения MVP

- План релизов заполнен до 29.04.2026 → дальше фолбэк «релиз по средам»
  (помечается в отчёте).
- Для задач «В разработке» планируется полный `developerEstimate` без вычета
  уже отработанного.
- Связи-блокеры, возвраты по замечаниям, fast-track — не учитываются
  (покрываются резервом).
- Мок-ёмкость не видит отпуска — до подключения HTTP-сервиса 1С.
